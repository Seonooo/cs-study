### 1. 왜 네트워크는 계층으로 나뉘어 있나? OSI 7계층과 TCP/IP 4계층은 왜 따로 존재하는가?

> 🎯 **면접 단골 질문**

<details><summary>▶ 힌트 보기</summary>

- 계층화 이유: 모듈성 / 재사용성 / 변경 격리 (separation of concerns)
- TCP가 IPv4·IPv6 어느 쪽 위에서도 동작하는 이유, HTTP가 TCP·QUIC 어느 쪽 위에서도 동작하는 이유와 연결
- OSI는 **이상적·교육용 참조 모델**, TCP/IP는 **실제 동작하는 구현 모델**. OSI의 L5/L6/L7은 TCP/IP에서 응용 계층 하나로 합쳐짐
- 트레이드오프: 계층화의 비용 — 오버헤드, 디버깅 난이도

</details>

### 2. 브라우저에 `https://example.com` 을 입력하고 Enter를 치면 무슨 일이 일어나는가? OSI 각 계층 관점에서 설명하라.

> 🎯 **백엔드 면접 1순위 단골 질문 (한국·해외 공통)**

<details><summary>▶ 힌트 보기</summary>

큰 흐름:

1. **DNS 조회** — `example.com` → IP 주소 (응용 계층 프로토콜이지만 동작은 UDP/TCP 위)
2. **TCP 연결** — 3-way handshake (L4)
3. **TLS 핸드셰이크** — 인증서 검증, 세션 키 합의
4. **HTTP 요청 송신** — 응용 계층(L7) 메시지 작성

각 계층에서 일어나는 일:

- L7 응용: HTTP 요청 메시지 작성
- L6/L5 (TLS): 암호화 + 세션 수립 (OSI 모델에서 정확한 위치는 모호)
- L4 전송: TCP 세그먼트 분할, 순서·재전송 보장
- L3 네트워크: IP 패킷 캡슐화, 라우팅
- L2 데이터링크: 프레임 캡슐화, MAC 기반 hop-by-hop 전달
- L1 물리: 비트로 변환되어 매체 전송

서버 응답은 역순으로 디캡슐화되어 브라우저에 도달.

</details>

### 3. L4 로드밸런서와 L7 로드밸런서는 무엇을 보고 라우팅을 결정하는가? 백엔드 인프라에서 언제 무엇을 쓰나?

> 🎯 **실무 핵심 — 클라우드/MSA 환경에서 매주 마주치는 결정**

<details><summary>▶ 힌트 보기</summary>

- **L4** (예: AWS NLB): IP/포트만 보고 결정 — 빠르고 프로토콜 무관, **TLS passthrough만 가능** (종료 불가)
- **L7** (예: AWS ALB, nginx, Envoy): HTTP 헤더·URL path·쿠키를 보고 결정 — 경로 기반 라우팅, A/B 테스트, sticky session, **TLS 종료 가능**
- 트레이드오프: L7은 CPU 비용 높음. 처리량 우선이면 L4, 유연성·관찰성·보안이 필요하면 L7
- 실무 패턴: NLB(L4) → ALB(L7) → 애플리케이션 으로 다단 구성하는 경우도 흔함

</details>

### 4. TCP는 L4, HTTP는 L7로 분류된다. 그렇다면 TLS는 어느 계층 프로토콜인가?

<details><summary>▶ 힌트 보기</summary>

- 정답이 하나가 아닌 흥미로운 질문 (면접에서 깊이를 가늠할 때 자주 등장)
- 자료마다 다름:
  - **L6 (Presentation)** — 암호화/복호화가 표현 계층 역할이라는 시각 (다수설)
  - **L5 (Session)** — TLS handshake가 세션을 수립하므로
  - **"L4와 L7 사이"** — 실무에서 가장 흔한 설명
- 핵심 통찰: OSI는 1980년대 설계된 **이상적 참조 모델**이고, 1990년대 이후 실제 프로토콜은 깔끔하게 맞지 않는다는 점을 이해하면 됩니다
- 답변 팁: "엄밀히는 모호하지만 일반적으로 L6로 분류되며, 실무적으로는 TCP와 HTTP 사이에 위치한다고 설명합니다"

</details>

### 5. HTTP/3는 왜 TCP가 아닌 UDP 위에 만들어졌나? 계층화 관점에서 설명하라.

> 🎯 **2026 트렌드 — Cloudflare·Google이 주도, 한국 주요 서비스도 도입 중**

<details><summary>▶ 힌트 보기</summary>

- HTTP/2의 한계: 단일 TCP 연결 위에 여러 스트림을 다중화했지만, **TCP 레벨의 HoL(Head-of-Line) blocking** 발생 — 패킷 1개 손실이 모든 스트림을 멈춤
- TCP 자체를 바꿀 수 없는 이유: OS 커널 + 전 세계 라우터/방화벽이 이미 TCP에 최적화 → **변경 비용 무한대**
- 해결책 = **QUIC**: UDP 위에 신뢰성·순서보장·혼잡제어를 **유저 스페이스에서 직접 구현** → TCP를 우회하면서 빠르게 진화 가능
- 효과: 0-RTT 재연결, 스트림별 독립적 손실 회복, 모바일 IP 변경 시에도 연결 유지(Connection Migration)
- 계층화 통찰: "필요하면 더 낮은 계층을 우회하고 상위에서 다시 만든다" — 계층화의 유연성이 만든 진화

</details>
