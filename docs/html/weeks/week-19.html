Spring Boot — Auto-config·@Transactional·Actuator

framework 신설 주차 slug: spring-boot-tx

Spring Boot의 자동 구성이 어떻게 동작하고 어떻게 디버깅하는지, @Transactional이 안 먹는 함정 케이스가 무엇인지를 짚는다. 2026 면접에서 auto-config 디버깅 능력이 강조됨.

학습 목표

  1. @SpringBootApplication의 3 어노테이션 분해
  2. Auto-configuration 디버깅 (--debug, AutoConfigurationReport)
  3. @Transactional이 안 먹는 케이스 (self-invocation, private, checked exception)
  4. 트랜잭션 전파 속성과 적용 사례
  5. Actuator 엔드포인트로 운영 가시성 확보

핵심 키워드

  • @SpringBootApplication
  • Auto-configuration · @Conditional
  • @Transactional 프록시 함정
  • 전파 (Propagation)
  • Actuator

딥다이브

  • @SpringBootApplication 분해 — @Configuration + @EnableAutoConfiguration + @ComponentScan의 합성 어노테이션.
  • Auto-config 동작 — META-INF/spring/...AutoConfiguration.imports의 후보를 @Conditional(클래스 존재·빈 부재·프로퍼티)로 선별 적용.
  • 디버깅 — --debug로 CONDITIONS EVALUATION REPORT 출력 → 어떤 auto-config가 matched/negative인지 확인.
  • @Transactional 함정 — 같은 클래스 self-invocation(프록시 우회), private/final 메서드, 기본 롤백은 unchecked 예외만(checked는 rollbackFor 명시).
  • 전파 속성 — REQUIRED(기본, 기존 트랜잭션 참여) · REQUIRES_NEW(새 트랜잭션, 독립 커밋) · NESTED(savepoint) 등.

토론 주제

Spring Boot가 가린 "마법"을 어디까지 이해해야 하는가?

면접 단골 Q&A

@SpringBootApplication은 어떤 어노테이션으로 구성되나요?

@Configuration(빈 정의 소스) + @EnableAutoConfiguration(조건부 자동 구성) + @ComponentScan(컴포넌트 탐색).

@Transactional이 동작하지 않는 케이스는?

같은 클래스 내부 self-invocation(프록시 우회), private/final 메서드, 기본 설정에서 checked exception 발생 시 롤백 안 됨(rollbackFor 필요).

REQUIRED와 REQUIRES_NEW의 차이는?

REQUIRED는 기존 트랜잭션에 참여(없으면 생성). REQUIRES_NEW는 항상 새 트랜잭션을 만들어 독립적으로 커밋/롤백 → 로깅·이력처럼 본 트랜잭션과 분리해야 할 때.
