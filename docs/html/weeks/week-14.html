트랜잭션과 격리 수준

database slug: transaction-isolation

ACID 4속성과 격리 수준 4단계, 그리고 각 단계가 어떤 이상 현상을 허용/방지하는지를 매핑할 수 있어야 한다. MVCC와 락 기반의 차이도 면접 단골 영역.

학습 목표

  1. ACID 각 속성의 의미
  2. 격리 수준과 이상 현상 매핑
  3. MVCC 동작 원리
  4. MySQL InnoDB vs PostgreSQL Repeatable Read 차이
  5. Write Skew와 SSI

핵심 키워드

  • ACID
  • 격리 수준 4단계
  • MVCC vs 락 기반
  • Gap Lock
  • Snapshot Isolation
  • Write Skew · SSI

딥다이브

  • 이상 현상 매핑 — Read Uncommitted(dirty 허용) → Read Committed(dirty 차단) → Repeatable Read(non-repeatable 차단) → Serializable(phantom까지 차단).
  • MVCC — 행 버전을 유지해 읽기가 쓰기를 막지 않음. 트랜잭션은 시작 시점의 스냅샷으로 일관된 읽기.
  • InnoDB의 Gap Lock — Repeatable Read에서 phantom을 방지하기 위해 인덱스 범위에 락 추가.
  • Write Skew — Snapshot Isolation에서도 발생 가능한 이상. 두 트랜잭션이 서로 다른 행을 읽고 쓸 때 비즈니스 불변식이 깨짐.
  • SSI — PostgreSQL의 Serializable. SI 위에 의존 그래프 추적으로 직렬화 가능성 깨지면 abort.

실무에서 격리 수준을 어떻게 선택하는가? Read Committed가 사실상 표준이 된 이유는?

면접 단골 Q&A

ACID 각 속성을 설명해 주세요.

Atomicity(전부 성공 또는 전부 롤백), Consistency(불변식 유지), Isolation(동시 실행이 직렬과 동일한 효과), Durability(커밋 후 영속).

격리 수준과 이상 현상의 매핑은?

RU: dirty read · NR · phantom 모두 가능. RC: dirty 차단. RR: NR까지 차단(MySQL은 phantom도 gap lock으로 차단). SER: 전부 차단.

MVCC와 락 기반의 차이는?

락 기반은 읽기가 쓰기를 막음(성능 저하). MVCC는 버전을 두어 읽기는 락 없이 진행, 쓰기 충돌만 락/abort로 해결.
