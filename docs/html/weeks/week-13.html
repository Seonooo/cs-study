쿼리 최적화와 EXPLAIN

database 신설 주차 slug: query-optimization

EXPLAIN 결과를 읽어 병목을 짚어내고, 옵티마이저가 인덱스를 안 타는 이유를 진단할 수 있어야 한다. 조인 알고리즘 3종(NL/Hash/Merge)의 적합 상황까지가 면접에서 깊게 들어가는 단골.

학습 목표

  1. EXPLAIN 결과 읽고 병목 파악
  2. 옵티마이저가 인덱스를 안 타는 케이스 진단
  3. 조인 알고리즘별 선택 기준
  4. Slow query log 활용
  5. 쿼리 리라이팅 전략

핵심 키워드

  • EXPLAIN · EXPLAIN ANALYZE
  • 옵티마이저 통계
  • 인덱스 미사용
  • Slow query
  • NL · Hash · Merge Join

딥다이브

  • EXPLAIN 핵심 컬럼 — type(접근 방식: ref/range/ALL) · key(쓴 인덱스) · rows(예상 행 수) · Extra(Using filesort/temporary 등).
  • 인덱스 안 타는 케이스 — 컬럼에 함수/연산, 타입 불일치, 선택도가 낮은 인덱스, 옵티마이저 통계가 오래된 경우.
  • 조인 알고리즘 — NL(작은 외부 테이블+인덱스 활용), Hash(대용량 + 인덱스 없음), Sort-Merge(정렬되어 있거나 정렬 비용 감수).
  • Slow query 활용 — 임계 설정(예: 1초), EXPLAIN ANALYZE로 실제 실행 비용 확인, 인덱스 후보 도출.
  • 쿼리 리라이팅 — IN(서브쿼리) → JOIN, OR 분기 → UNION ALL, 페이지네이션은 OFFSET 대신 keyset.

토론 주제

ORM이 만들어내는 쿼리를 어디까지 신뢰할 것인가? 어디서부터 SQL을 직접 쓰는가?

면접 단골 Q&A

EXPLAIN에서 가장 먼저 보는 것은?

type과 rows. type이 ALL(풀스캔)이면 인덱스 미사용 가능성, rows가 예상보다 크면 통계 또는 카디널리티 추정 오류.

옵티마이저가 인덱스를 안 타는 흔한 이유는?

WHERE 컬럼에 함수/연산 적용, 타입 불일치(암묵적 형변환), 통계가 오래되어 풀스캔이 더 싸 보임, 결과 행이 테이블의 큰 비중을 차지(인덱스 비용이 더 큼).

Hash Join은 언제 유리한가요?

두 테이블이 크고 조인 키에 인덱스가 없을 때. 한 쪽을 메모리 해시 테이블로 만들고 다른 쪽을 한 번만 스캔.
