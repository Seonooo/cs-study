PHASE 2 · OS · WEEK 07 · 신설

가상 스레드·코루틴·동시성 모델

os 신설 주차 (2026 트렌드) slug: virtual-thread-coroutine

Thread-per-request의 메모리·스위칭 한계가 어떻게 가상 스레드와 코루틴으로 해결되었는지, M:N 스케줄링이 어떻게 처리량을 끌어올리는지를 이해한다. Java 21의 Virtual Thread는 2026 면접 단골.

학습 목표

  1. Thread-per-request 모델의 한계
  2. Virtual Thread가 OS 스레드 비용을 줄이는 메커니즘
  3. Goroutine M:N 스케줄링
  4. Coroutine vs Virtual Thread 차이
  5. 멀티프로세스 / 멀티스레드 / 비동기 I/O / 가상스레드 선택 기준

핵심 키워드

  • Java Virtual Thread (Project Loom)
  • Goroutine M:N
  • Kotlin Coroutine
  • Reactive / async I/O
  • Carrier thread

딥다이브

  • Thread-per-request 한계 — OS 스레드 ≈ 수 MB 스택 + 컨텍스트 스위칭 비용. 수만 동시 연결에서 메모리·스위칭이 병목.
  • Virtual Thread 메커니즘 — JVM이 가벼운 스레드를 생성, blocking 시점에 carrier thread에서 unmount되고 다른 가상 스레드가 mount. 사용자 코드는 동기 스타일 그대로.
  • Goroutine M:N — Go 런타임이 M개의 goroutine을 N개의 OS 스레드에 스케줄. work-stealing 큐 + 시스템콜 시 스레드 분리(handoff).
  • Coroutine vs Virtual Thread — 코루틴은 suspend point가 명시적(언어 키워드), 가상 스레드는 일반 blocking 호출이 자동 yield. 모델은 비슷하나 통합 방식 차이.

토론 주제

Virtual Thread가 Reactive를 대체할 수 있는가? 어떤 영역은 여전히 Reactive가 유리한가?

면접 단골 Q&A

Virtual Thread와 일반 OS 스레드의 차이는?

OS 스레드는 커널이 관리(스택 MB 단위, 스위칭 시 모드 전환). 가상 스레드는 JVM이 관리(스택 동적, blocking 시 carrier에서 분리되어 다른 가상 스레드에 양보).

M:N 스케줄링이 무엇이고 왜 효율적인가?

M개의 사용자 레벨 태스크를 N개의 OS 스레드에 매핑(M ≫ N). 컨텍스트 스위칭이 유저 공간에서 일어나 비용 ↓, 한 태스크의 blocking이 전체를 막지 않도록 런타임이 핸드오프.

동시성 모델 4가지 — 언제 무엇을 쓸 것인가?

격리·신뢰성 우선이면 멀티프로세스, CPU 집약 + 공유 상태가 적으면 멀티스레드, I/O 중심 고동시성이면 비동기 I/O / 가상스레드.
