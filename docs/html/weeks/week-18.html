JPA — 영속성 컨텍스트와 N+1

framework 신설 주차 slug: jpa-persistence

영속성 컨텍스트가 주는 5가지 이점과 엔티티 생명주기를 이해하고, N+1 문제가 언제 발생하며 어떻게 해결하는지를 설명할 수 있어야 한다. 한국 백엔드 면접의 핵심 단골.

학습 목표

  1. 영속성 컨텍스트가 주는 5가지 이점
  2. 엔티티 생명주기 상태 전이
  3. 더티 체킹 동작 원리
  4. N+1 문제 발생 시점과 해결책 (Fetch Join, @EntityGraph, batch size)
  5. LazyInitializationException 발생 조건과 회피

핵심 키워드

  • 영속성 컨텍스트 5이점
  • 엔티티 생명주기
  • 더티 체킹
  • N+1 · Fetch Join
  • @EntityGraph · batch size
  • LazyInitializationException

딥다이브

  • 영속성 컨텍스트 5이점 — 1차 캐시 · 동일성 보장 · 트랜잭션 쓰기 지연 · 변경 감지(더티 체킹) · 지연 로딩.
  • 엔티티 생명주기 — transient(new) → persistent(persist/find) → detached(detach/close) → removed(remove). 영속 상태일 때만 더티 체킹·지연 로딩 동작.
  • 더티 체킹 — 영속 컨텍스트가 엔티티 스냅샷을 보관, 트랜잭션 커밋 시점에 변경 비교 후 자동 UPDATE.
  • N+1 — 연관 엔티티를 1건 조회 후 N건의 추가 쿼리. 즉시 로딩에서도 발생. 해결: Fetch Join(컬렉션은 페이징 한계), @EntityGraph, default_batch_fetch_size로 IN 절 묶음.
  • LazyInitializationException — 영속성 컨텍스트(트랜잭션) 종료 후 lazy 프록시 접근. OSIV·DTO 변환 시점 조정으로 회피.

토론 주제

JPA를 쓰면서 SQL을 모르면 위험한가? 어디까지 추상화를 신뢰할 것인가?

면접 단골 Q&A

영속성 컨텍스트의 이점을 설명해 주세요.

1차 캐시(같은 트랜잭션 내 동일 조회는 DB 미접근), 동일성 보장(==), 쓰기 지연(SQL 모았다 flush), 변경 감지(자동 UPDATE), 지연 로딩(필요 시점에 조회).

N+1 문제가 무엇이고 어떻게 해결하나요?

연관 엔티티를 N건 조회할 때 쿼리가 1+N개 발생. Fetch Join으로 한 번에, 컬렉션 페이징이 필요하면 batch size로 IN 절 묶음, 또는 @EntityGraph.

더티 체킹은 어떻게 동작하나요?

영속 컨텍스트가 엔티티 최초 상태의 스냅샷을 보관하고, flush/commit 시점에 현재 값과 비교해 변경된 필드만 UPDATE 생성. 별도 save 호출 불필요.
