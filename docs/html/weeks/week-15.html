PHASE 3 · DATABASE · WEEK 15

NoSQL, 분산 데이터베이스

database slug: nosql-distributed

CAP / PACELC로 분산 DB의 트레이드오프를 분류하고, 복제 방식과 합의 알고리즘이 어떤 보장을 주는지 설명한다. MSA 데이터 일관성 패턴(Saga·Outbox)까지가 면접 범위.

학습 목표

  1. CAP과 PACELC로 시스템 분류
  2. NoSQL 종류별 사용 사례
  3. 복제 방식 트레이드오프
  4. Eventual Consistency와 Quorum
  5. Saga 패턴과 Outbox

핵심 키워드

  • CAP · PACELC
  • KV · Document · Column-family · Graph
  • 단일/다중/리더리스 복제
  • Quorum · Vector clock
  • Saga · Outbox

딥다이브

  • CAP vs PACELC — CAP은 파티션 발생 시 C vs A. PACELC는 평시(E)에서 L(지연) vs C(일관성)까지 포함해 더 현실적 분류.
  • NoSQL 4종 — KV(Redis, DynamoDB), Document(MongoDB), Column-family(Cassandra, HBase), Graph(Neo4j).
  • 복제 방식 — 단일 리더(쓰기 단순, 리더 장애가 SPOF) / 다중 리더(WAN 다지역) / 리더리스(Dynamo 스타일, quorum 기반).
  • Saga 패턴 — 분산 트랜잭션을 일련의 로컬 트랜잭션 + 보상 트랜잭션으로 대체. choreography vs orchestration.
  • Outbox 패턴 — DB와 메시지 큐의 듀얼 라이트 문제 → 같은 트랜잭션에 outbox 테이블 기록 후 이벤트 발행.

토론 주제

마이크로서비스에서 데이터 일관성을 어떻게 보장할 것인가?

면접 단골 Q&A

CAP과 PACELC의 차이는?

CAP은 파티션 시점만 다룸. PACELC는 평시(E)에서 L과 C 트레이드오프까지 명시 → 더 현실적인 분류(예: Cassandra = PA/EL).

리더리스 복제의 quorum은 어떻게 동작하나요?

N개 노드에 쓰기를 W개에 확인, 읽기는 R개에서 → W+R>N이면 강한 일관성. 일관성과 가용성/지연을 R·W 조정으로 절충.

Saga 패턴은 무엇인가요?

분산 트랜잭션 대신 일련의 로컬 트랜잭션. 중간 실패 시 이전 단계를 보상(compensating) 트랜잭션으로 되돌림. orchestration 또는 choreography 방식.
