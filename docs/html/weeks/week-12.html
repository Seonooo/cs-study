인덱스와 B+Tree

database slug: index-bptree

대부분의 RDBMS가 왜 B+Tree를 인덱스 자료구조로 채택했는지, 클러스터드 인덱스 구조가 PK 설계에 어떤 영향을 주는지, 복합 인덱스의 컬럼 순서를 어떻게 결정하는지를 다룬다.

학습 목표

  1. DB가 B+Tree를 쓰는 이유
  2. InnoDB PK 설계 가이드 (클러스터드 영향)
  3. 복합 인덱스 컬럼 순서 결정
  4. 커버링 인덱스로 쿼리 최적화
  5. B+Tree와 해시 인덱스의 트레이드오프

핵심 키워드

  • B-Tree vs B+Tree
  • Clustered / Non-clustered
  • 복합 인덱스 컬럼 순서
  • 커버링 인덱스
  • 해시 인덱스

딥다이브

  • B+Tree가 표준인 이유 — 모든 데이터가 leaf에만 있어 범위 스캔에 유리, 리프 노드 연결 리스트로 순회 가능. fan-out이 커서 디스크 I/O 최소화.
  • 클러스터드 vs 논클러스터드 — InnoDB의 PK는 클러스터드(데이터 자체가 PK 순). 보조 인덱스는 PK 값을 leaf로 가짐 → PK가 길면 모든 보조 인덱스가 비대해짐.
  • 복합 인덱스 순서 — 등호 조건이 자주 오는 컬럼을 앞에. 범위 조건은 뒤에 두어야 다음 컬럼이 인덱스 활용 가능.
  • 커버링 인덱스 — SELECT/WHERE/ORDER BY가 모두 인덱스 안의 컬럼만 쓰면 테이블 접근 없이 인덱스만으로 응답.

토론 주제

인덱스를 추가했는데 오히려 느려진 사례 — 왜 일어나는가?

면접 단골 Q&A

DB는 왜 B-Tree 대신 B+Tree를 쓰나요?

B+Tree는 leaf에만 데이터가 있어 fan-out이 커지고, leaf 간 연결 리스트로 범위 스캔이 효율적. 디스크 기반 인덱스에 최적.

InnoDB PK는 왜 짧고 단조 증가하는 게 좋은가요?

클러스터드라 모든 보조 인덱스 leaf가 PK 값을 갖기 때문에 PK가 길면 인덱스 전체가 비대해짐. 무작위 PK는 페이지 분할/조각화 유발.

복합 인덱스 (A, B, C)에서 어떤 쿼리가 인덱스를 타나요?

A · A+B · A+B+C 조건이 prefix matching으로 활용. WHERE B = ? 만 있으면 인덱스를 못 탐.
