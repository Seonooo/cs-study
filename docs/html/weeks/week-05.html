웹 인증 — 세션·쿠키·JWT·OAuth2

network 신설 주차 slug: web-auth-session

세션/쿠키 기반과 토큰 기반(JWT) 인증의 차이를 알고, OAuth2 Authorization Code Flow의 단계를 설명할 수 있어야 한다. CSRF·XSS 방어 패턴까지가 면접 단골 세트.

학습 목표

  1. 세션 기반 vs 토큰 기반 인증 선택 기준
  2. JWT의 장점과 위험 (탈취·만료·블랙리스트)
  3. OAuth2 Authorization Code Flow 단계
  4. CSRF / XSS 방어 패턴
  5. SSO·OIDC 동작 흐름

핵심 키워드

  • Session vs Cookie
  • JWT
  • OAuth2 4 grant

딥다이브

  • JWT 구조 — header.payload.signature. 서명 검증으로 무결성, payload는 base64url 인코딩(암호화 아님 — 민감정보 금지).
  • JWT 위험 — 탈취 시 만료 전까지 유효. 블랙리스트·짧은 TTL + Refresh Token 회전으로 완화.
  • OAuth2 Authorization Code + PKCE — 리다이렉트 기반, code_verifier로 공개 클라이언트(SPA/모바일)의 인가 코드 탈취 방어.
  • CSRF 방어 — SameSite 쿠키, CSRF 토큰, Origin/Referer 검증.
  • XSS 방어 — 출력 이스케이프, CSP 헤더, HttpOnly 쿠키로 토큰 저장.
  • OIDC — OAuth2 위에 ID Token(JWT) 추가 → 인증(authentication)까지 표준화.

면접 단골 Q&A

세션과 쿠키의 차이는?

쿠키는 브라우저 저장소·전송 메커니즘. 세션은 서버에 사용자 상태를 저장하고 쿠키로 식별자(Session ID)만 전달하는 인증 패턴.

JWT의 위험과 완화책은?

탈취 시 만료 전까지 유효 → 짧은 액세스 토큰 + Refresh Token 회전, 블랙리스트, HttpOnly 쿠키, 서명 알고리즘 검증(특히 none 방어).

CSRF와 XSS는 어떻게 다른가?

CSRF는 사용자가 인증된 상태에서 의도치 않은 요청 강제 → SameSite·CSRF 토큰. XSS는 스크립트 주입 → 출력 이스케이프·CSP.
