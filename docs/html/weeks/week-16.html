Spring IoC·DI·Bean

framework 신설 주차 slug: spring-ioc-di

Spring의 출발점인 IoC 컨테이너와 DI 3방식, Bean 생명주기·스코프, 그리고 순환 의존성이 왜 생기고 어떻게 해결하는지를 다룬다. 생성자 주입이 권장되는 이유는 면접 단골.

학습 목표

  1. IoC와 DI 개념과 도입 이유
  2. 생성자 주입이 권장되는 이유
  3. Bean 생명주기 단계와 콜백 (@PostConstruct, @PreDestroy)
  4. Singleton vs Prototype 선택
  5. 순환 의존성 발생 원인과 해결

핵심 키워드

  • IoC Container
  • BeanFactory · ApplicationContext
  • DI 3방식 (Constructor · Setter · Field)
  • Bean Lifecycle
  • Bean Scope
  • 순환 의존성

딥다이브

  • IoC의 의미 — 객체 생성·소멸·연결의 제어권을 프레임워크로 역전. 사용자는 "무엇이 필요한지"만 선언.
  • 생성자 주입 권장 이유 — 불변성(final), 누락 시 컴파일 단계 인지, 순환 의존성 조기 탐지, 테스트 용이.
  • Bean Lifecycle — 인스턴스화 → 의존성 주입 → BeanNameAware/BeanFactoryAware → BeanPostProcessor(before) → InitializingBean/@PostConstruct → 사용 → @PreDestroy/DisposableBean.
  • Scope — singleton(컨테이너당 1개, 기본) / prototype(요청마다 새 인스턴스, 컨테이너가 destroy 책임지지 않음) / request · session · application(웹 전용).
  • 순환 의존성 — 생성자 주입 + 양방향 의존이면 BeanCurrentlyInCreationException. 해결: 설계 분리, @Lazy, 또는 Setter 주입.

토론 주제

Spring의 IoC 컨테이너 없이 우리 손으로 만들 때 무엇이 어려운가?

면접 단골 Q&A

IoC와 DI의 차이는?

IoC는 제어 흐름 역전이라는 일반 원칙. DI는 IoC를 구현하는 한 방법(의존성을 외부에서 주입). DI 외에도 Service Locator, Template Method 등이 IoC.

생성자 주입을 권장하는 이유는?

final 필드로 불변성 확보, 누락된 의존성을 컴파일/시작 시점에 발견, 순환 의존성을 조기 탐지, 테스트에서 new로 직접 주입 가능.

Singleton과 Prototype 중 무엇을 쓸지 어떻게 결정하나요?

상태 없는 서비스/리포지토리는 singleton(메모리 효율). 요청마다 별도 상태가 필요하면 prototype. prototype은 컨테이너가 destroy 안 하므로 자원 해제 책임을 직접 짐.
