PHASE 1 · NETWORK · WEEK 04

DNS, 로드밸런싱, CDN

network 다음 주차 slug: dns-lb-cdn

도메인 이름이 어떻게 IP로 변환되고, 트래픽이 여러 서버에 어떻게 분배되며, CDN이 어디서 캐싱해 응답을 빠르게 하는지의 큰 흐름을 잡는다.

학습 목표

  1. DNS 조회 흐름 (recursive vs iterative)
  2. TTL 트레이드오프와 마이그레이션 전략
  3. Sticky session vs Stateless 결정 기준
  4. Anycast 라우팅 원리
  5. CDN 캐시 전략 (origin shield, stale-while-revalidate)

핵심 키워드

  • DNS recursive / iterative
  • TTL 마이그레이션
  • Sticky session
  • Anycast
  • Origin Shield
  • stale-while-revalidate

딥다이브

  • DNS 계층 — Root → TLD → Authoritative. Resolver는 캐시·forwarder 역할.
  • TTL 마이그레이션 — 변경 직전 TTL을 미리 낮춰 전파 지연 최소화 후, 안정화 뒤 다시 올림.
  • Sticky vs Stateless — 세션 친화성이 필요하면 sticky, 무중단 확장이 우선이면 stateless + 외부 세션 저장소.
  • Anycast — 같은 IP를 여러 PoP에 광고 → BGP가 가장 가까운 노드로 라우팅. CDN·DNS에 광범위 활용.
  • CDN 캐싱 키 — URL + query + 헤더로 키 구성, vary 헤더와 cache-control 조합으로 정밀 제어.

토론 주제

글로벌 서비스를 출시한다면 어떤 순서로 인프라(DNS·LB·CDN)를 배치할 것인가?

면접 단골 Q&A

DNS 조회 흐름을 설명해 주세요.

클라이언트 → Recursive Resolver → Root → TLD → Authoritative → 응답 캐시 후 클라이언트. 각 단계에 TTL 기반 캐시.

Sticky session vs Stateless, 언제 무엇을 쓰나요?

레거시 세션 의존 앱이나 인메모리 캐시 사용 시 sticky가 단순. 클라우드 네이티브·오토스케일 환경에서는 외부 세션 저장소 + stateless가 표준.

CDN의 origin shield는 어떤 역할을 하나요?

엣지 노드들 사이에 중간 레이어를 두어 origin으로 가는 요청을 통합·중복 제거. cache miss 증폭을 줄여 origin 부하 보호.
