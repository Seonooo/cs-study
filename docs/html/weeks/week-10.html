파일 시스템과 I/O 모델

os slug: io-models

sync/async와 blocking/non-blocking 4사분면을 정확히 구분하고, select → poll → epoll → io_uring으로 이어지는 I/O 모델 진화의 동기를 설명할 수 있어야 한다.

학습 목표

  1. sync/async와 blocking/non-blocking 구분
  2. epoll 효율의 비밀
  3. ET (Edge Triggered) / LT (Level Triggered) 차이와 선택
  4. io_uring 등장 배경
  5. Node.js 단일 스레드 처리량

핵심 키워드

  • sync/async vs blocking/non-blocking
  • select · poll · epoll
  • ET vs LT
  • io_uring
  • Node.js Event Loop

딥다이브

  • 4사분면 — sync/blocking, sync/non-blocking(poll loop), async/blocking(드물), async/non-blocking(완전 비동기 콜백).
  • epoll의 효율 — select/poll은 매 호출마다 O(N) 순회. epoll은 커널이 fd 상태 변화를 큐로 알림 → O(1) 조회.
  • ET vs LT — LT: 데이터가 남아있는 한 계속 알림(안전, 간단). ET: 상태 변화 시 한 번만(빠름, non-blocking + 완전 소비 필요).
  • io_uring — submission/completion ring을 mmap으로 공유 → 시스템콜 없이 다중 I/O 제출/회수. 디스크·네트워크 모두 일관된 비동기.
  • Node.js — libuv 위에 단일 JS 스레드 + epoll/kqueue 이벤트 루프. CPU 작업이 들어오면 막힘 → worker_threads로 분리.

면접 단골 Q&A

sync/async와 blocking/non-blocking은 어떻게 다른가요?

blocking은 호출자가 결과를 기다리는지(제어 흐름 관점). sync는 결과를 어떻게 받는지(통지 방식). 둘은 독립 축이라 4사분면이 가능.

select와 epoll의 차이는?

select는 매번 fd_set을 커널에 복사하고 O(N) 순회. epoll은 fd 등록을 한 번 하고 커널이 이벤트 큐로 알림 → 고동시성에 유리.

io_uring이 epoll보다 무엇이 더 좋은가요?

epoll은 readiness만 알리고 read/write는 별도 syscall. io_uring은 submission/completion ring으로 syscall 자체를 줄이며, 네트워크·디스크 I/O를 일관된 비동기 인터페이스로.
