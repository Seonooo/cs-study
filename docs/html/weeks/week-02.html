TCP, UDP

network 완료 slug: tcp-udp

TCP의 신뢰성 메커니즘(핸드셰이크·흐름제어·혼잡제어)을 이해하고, UDP와 비교해 언제 무엇을 선택해야 하는지 판단할 수 있게 된다.

학습 목표

  1. 3-way / 4-way handshake 동작과 그 횟수인 이유
  2. TIME_WAIT 발생 원인과 해결책
  3. TCP vs UDP 선택 기준
  4. Connection Pool / Keepalive 의 역할

핵심 키워드

  • 3·4-way handshake
  • TIME_WAIT
  • 흐름 제어
  • 혼잡 제어 (BBR)
  • Connection Pool
  • Nagle
  • TCP Fast Open

딥다이브

  • TIME_WAIT가 필요한 이유 — 마지막 ACK 손실 대비 + 이전 연결의 지연 패킷이 새 연결을 오염시키지 않도록 (2MSL).
  • Nagle 알고리즘 — 작은 패킷 묶음 송신으로 효율 향상, 반대로 지연 민감 트래픽엔 TCP_NODELAY.
  • 혼잡 제어 진화 — Reno → CUBIC → BBR. BBR은 대역폭과 RTT를 측정해 손실 없이 혼잡 신호 탐지.
  • QUIC가 UDP 위에 만들어진 이유 — TCP의 HoL blocking·핸드셰이크 RTT를 사용자 공간에서 재설계.

토론 주제

실시간성이 중요한 게임/영상에서 TCP vs UDP 선택 기준은?

면접 단골 Q&A

TCP와 UDP의 차이는?

TCP는 연결 지향·신뢰성·순서 보장, UDP는 비연결·고속·헤더 8바이트. 신뢰성이 필요 없거나 직접 구현하려는 경우 UDP.

왜 3-way이고 4-way handshake인가?

연결 수립은 양방향 SYN/ACK을 한 번씩 교환하면 3개로 충분하지만, 종료는 한쪽이 먼저 끝나도 다른 쪽이 데이터 송신을 끝낼 시간이 필요해 FIN/ACK가 각각 분리되어 4개.

Connection Pool과 Keepalive의 역할은?

매 요청마다 TCP 핸드셰이크(+ TLS) 비용을 피하기 위해 연결을 재사용. 풀 크기 튜닝으로 동시성과 자원 사이 균형.
