메모리 관리와 GC

os slug: memory-gc

OS의 가상 메모리·페이징부터 JVM의 GC 진화(Serial → G1 → ZGC)까지 메모리 계층 전체를 한 줄로 꿰고, Stop-the-world 시간이 어떻게 줄어왔는지 설명할 수 있어야 한다.

학습 목표

  1. 가상 메모리 목적과 페이징 동작
  2. TLB miss vs Page Fault 비용 계층
  3. 캐시 친화적 코드 작성 원칙
  4. JVM 메모리 영역과 GC 알고리즘 진화 (Serial → G1 → ZGC)
  5. Stop-the-world 시간이 짧아진 메커니즘

핵심 키워드

  • 가상 메모리 / 페이징
  • TLB miss · Page Fault
  • JVM 메모리 영역
  • ZGC · Shenandoah · G1
  • 캐시 라인

딥다이브

  • 가상 메모리 목적 — 프로세스 간 격리 + 물리 메모리 추상화 + 디스크로 swap.
  • 비용 계층 — 레지스터 ≪ L1 캐시 ≪ TLB hit ≪ TLB miss ≪ Page Fault(디스크). 한 단계 내려갈 때마다 수십~수만 배.
  • 캐시 친화적 코드 — 순차 접근, 작은 데이터 구조, false sharing 회피.
  • GC 진화 — Serial(STW 전체) → Parallel(병렬 STW) → CMS(concurrent mark) → G1(영역 분할, 예측 가능 GC) → ZGC/Shenandoah(STW < 10ms, colored pointer/load barrier).

토론 주제

GC 언어와 수동 메모리 관리 언어의 트레이드오프 — 어디까지가 자동화의 비용인가?

면접 단골 Q&A

가상 메모리는 왜 필요한가요?

프로세스 간 격리, 물리 메모리보다 큰 주소 공간 사용, 페이지 단위 보호·공유. 페이징을 통해 disk swap도 가능.

TLB miss와 Page Fault의 차이는?

TLB miss는 페이지 테이블에서 매핑을 찾는 비용(보통 수십~수백 ns). Page Fault는 페이지 자체가 메모리에 없어 디스크에서 가져와야 하는 비용(ms 단위).

ZGC가 STW를 어떻게 10ms 이하로 줄였나요?

colored pointer로 객체 상태를 포인터 비트에 인코딩하고, load barrier로 동시 마킹·재배치를 지원. 대부분의 GC 작업이 애플리케이션 스레드와 동시에 실행.
