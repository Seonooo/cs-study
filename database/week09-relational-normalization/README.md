### 1. 1NF / 2NF / 3NF / BCNF는 각각 무엇이며 어떤 이상 현상을 방지하나?

> 🎯 **면접 단골 — DB 가장 기본**

<details><summary>▶ 힌트 보기</summary>

**3가지 이상 현상(Anomaly):**

- **Insert anomaly** — 무관한 정보가 없으면 새 데이터를 넣을 수 없음
- **Update anomaly** — 같은 데이터가 여러 행에 있어 일부만 갱신되면 모순
- **Delete anomaly** — 한 행 삭제로 무관한 정보가 함께 사라짐

**정규화 단계:**

| Form | 조건 | 방지하는 문제 |
|------|------|-------------|
| **1NF** | 모든 속성이 원자값(atomic), 중복 행 없음 | 다중 값/리스트로 인한 검색 불가 |
| **2NF** | 1NF + **부분 함수 종속 제거** (복합키의 일부에만 종속되는 비키 속성 X) | Insert/Update anomaly |
| **3NF** | 2NF + **이행 함수 종속 제거** (비키 → 비키 종속 X) | Update anomaly (이행 종속) |
| **BCNF** | 3NF + **모든 결정자가 슈퍼키** | 3NF가 못 잡는 일부 anomaly |

**예시 — 학생 수강 테이블:**

```
[학번, 과목, 교수, 교수실, 점수]
- (학번, 과목) → 점수 (전체 키 종속) ✓
- 과목 → 교수, 교수 → 교수실 (이행 종속!)
```

→ 3NF 위반. 분리 필요:
- `수강[학번, 과목, 점수]`
- `과목[과목, 교수]`
- `교수[교수, 교수실]`

**3NF vs BCNF 차이:**

- 3NF는 "비키 속성이 키에만 종속"
- BCNF는 "**모든** 결정자가 슈퍼키여야 함"
- 3NF 만족하나 BCNF 위반 사례는 드물지만 존재

**핵심 통찰:** 정규화는 **데이터의 독립성**을 높이는 과정. 한 사실이 한 곳에만 저장되도록.

</details>

### 2. 함수 종속성(Functional Dependency)이란 무엇이며 왜 정규화의 핵심인가? 부분 종속과 이행 종속을 예시로 설명하라.

> 🎯 **깊이 — 정규화의 이론적 토대**

<details><summary>▶ 힌트 보기</summary>

**정의:**

`X → Y` (X가 Y를 함수적으로 결정한다) — X의 값이 같으면 Y의 값도 같음.

예: `학번 → 이름` (학번이 같으면 이름도 같음)

**왜 정규화의 핵심?**

- 정규화의 모든 단계(2NF, 3NF, BCNF)가 함수 종속성으로 정의됨
- "어떤 사실이 어디에 저장되어야 하는가"를 함수 종속성이 결정

**부분 함수 종속 (Partial Dependency):**

- 복합 키 `(A, B)` 에서 비키 속성이 **A 또는 B에만 종속**
- 예: 키 `(학번, 과목)` 에서 `학번 → 학생이름`
- 이름은 학번에만 종속 → 같은 학생의 이름이 여러 행에 중복

→ 2NF 위반. 학생이름은 별도 테이블로 분리.

**이행 함수 종속 (Transitive Dependency):**

- A → B 이고 B → C 이면 A → C 가 이행 종속
- 비키 → 비키 종속이 핵심 문제
- 예: `학번 → 학과 → 학과사무실` 에서 학과사무실은 학과를 통해 간접 종속

→ 3NF 위반. 학과 정보는 별도 테이블로.

**자명한(trivial) 종속성:**

- `(A, B) → A` 같은 경우 — 자기 자신에 종속. 무시
- 비자명한 종속성만 정규화 대상

**다치 종속성(Multivalued Dependency):**

- A 하나에 B 집합이 종속, A 하나에 C 집합이 종속, B와 C는 독립
- 4NF에서 다룸 (실무에선 거의 안 다룸)

**실무 적용:**

- ER 다이어그램 그릴 때 함수 종속성 분석이 자연스럽게 일어남
- "이 정보를 어디에 둘까?" → "이 속성이 어떤 키에 종속되는가?" 로 답

</details>

### 3. 정규화와 반정규화의 트레이드오프는? 실무에서 3NF까지만 가는 이유와 언제 반정규화를 선택해야 하는가?

> 🎯 **면접 단골 + 실무**

<details><summary>▶ 힌트 보기</summary>

**정규화의 장점:**

- **데이터 무결성** — 한 사실이 한 곳에만 저장 → 갱신 모순 없음
- **저장 공간 절약**
- **변경 관리 쉬움** — 한 행 수정으로 모든 곳에 반영

**정규화의 단점:**

- **JOIN 비용** — 흩어진 정보를 합치려면 JOIN 필요 (대용량에서 비싸짐)
- **쿼리 복잡도** — 여러 테이블 조인하는 SQL 작성 부담
- 분산 환경에서 cross-shard JOIN은 거의 불가능

**반정규화의 이유:**

- 읽기 성능 (특히 대용량 read-heavy)
- 분석 쿼리 단순화
- 분산 / 샤딩된 환경

**왜 실무에서 3NF까지?**

- 3NF로도 대부분의 anomaly 방지 가능
- BCNF는 이론적으로 더 엄격하지만 분해 시 함수 종속성 보존이 안 될 수 있음
- 4NF/5NF는 실무 빈도 매우 낮음
- 결과: **"3NF 정도면 충분, 측정 후 필요한 곳만 반정규화"**

**언제 반정규화?**

| 상황 | 패턴 |
|------|------|
| 빈번한 JOIN으로 성능 저하 | 자주 함께 읽는 컬럼을 같은 테이블에 (e.g., user 테이블에 카운터 추가) |
| 통계/집계 매번 계산 비용 | summary table, materialized view |
| 분석/리포트 시스템 (OLAP) | star schema, snowflake schema |
| 분산 DB / 마이크로서비스 | 서비스마다 자체 데이터 사본 (eventual consistency) |
| 캐시 히트율 높이기 | 자주 변하지 않는 값을 인라인 |

**실무 격언:**

- **"Normalize until it hurts, denormalize until it works."**
- 정규화된 스키마 → 반정규화는 추가 컬럼/테이블로 가능
- 반정규화부터 시작하면 데이터 모순이 누적되어 정리 어려움

**반정규화의 비용:**

- update 시 여러 곳 동시 갱신 (애플리케이션 또는 DB 트리거)
- 동기화 깨질 위험 → 모니터링 + 재계산 배치 필요

</details>

### 4. OLTP와 OLAP의 차이는? 각각에 적합한 정규화 정책과 Star schema는 어떻게 동작하나?

> 🎯 **백엔드 + 데이터 엔지니어링 — 분석 시스템 설계**

<details><summary>▶ 힌트 보기</summary>

**OLTP (Online Transaction Processing):**

- 트랜잭션 중심 — 주문 생성, 회원 가입 등
- 작은 단위 read/write 가 빠르게 반복
- 일관성·무결성 최우선
- 정규화 우선 (3NF 정도)
- 예: MySQL, PostgreSQL 위 일반 백엔드

**OLAP (Online Analytical Processing):**

- 분석/집계 중심 — 매출 리포트, 사용자 행동 분석
- 대량 데이터를 한 번에 스캔, 복잡한 집계
- 쓰기는 batch (ETL)
- 반정규화 활용 → JOIN 줄여 스캔 효율↑
- 예: Snowflake, BigQuery, Redshift, ClickHouse

**비교:**

| 항목 | OLTP | OLAP |
|------|------|------|
| 워크로드 | 짧은 트랜잭션 | 긴 분석 쿼리 |
| 데이터 변경 빈도 | 매우 높음 | 낮음 (batch) |
| 행/열 지향 | 행 지향 | **열 지향(columnar)** |
| 정규화 | 3NF | 반정규화 (star schema) |
| 응답 시간 | ms | s ~ min |
| 대표 사용자 | 애플리케이션 | 분석가 / BI 도구 |

**Star Schema:**

- 중앙의 **Fact 테이블** + 주변의 **Dimension 테이블**
- Fact: 측정값 + dimension key (예: 매출 fact = 날짜, 상품, 고객, 매장 + 금액)
- Dimension: 차원 정보 (날짜, 상품, 고객, 매장)
- JOIN 1단계만 → 단순·빠른 분석

```
       [날짜 dim]
          |
[고객] -- [매출 FACT] -- [상품]
          |
       [매장 dim]
```

**Snowflake Schema:**

- Star + dimension 테이블도 정규화 (예: 상품 → 카테고리 → 부서)
- 더 정규화되어 저장 공간↓, JOIN 단계↑
- 일반적으로 Star schema 가 분석 성능 우수

**Lambda / Kappa 아키텍처 (실무):**

- OLTP DB → CDC(Change Data Capture) / Kafka → OLAP 시스템 으로 복제
- 두 시스템을 분리해서 각각 최적화

**실무 통찰:**

- 일반 백엔드 = OLTP, 분석 = 별도 OLAP 시스템
- 한 DB에서 둘 다 하려면 read replica + materialized view 활용
- "DB 하나에 다 넣지 마라" — 워크로드별로 다른 도구

</details>

### 5. 자연 키와 대리 키(surrogate key)의 차이는? 분산 환경에서 auto-increment / UUID v4 / UUID v7 / Snowflake ID 중 어떤 걸 선택해야 하나?

> 🎯 **백엔드 실무 + 2026 트렌드**

<details><summary>▶ 힌트 보기</summary>

**자연 키 (Natural Key):**

- 도메인에 이미 존재하는 식별자 (주민번호, 이메일, ISBN 등)
- 의미 있고 외부에서 알기 쉬움
- 단점: 도메인 규칙 변경 시 키도 변경 (이메일 변경 등) → FK 갱신 폭발

**대리 키 (Surrogate Key):**

- DB가 생성한 인공 식별자 (auto-increment, UUID 등)
- 도메인과 무관 → 변경 안전
- 단점: 의미 없음, 외부 노출 시 별도 표현 필요

**실무: 대부분 대리 키 + 자연 키는 unique 인덱스로 보호**

**키 전략 비교:**

| 키 종류 | 장점 | 단점 | 적합한 경우 |
|--------|------|------|-----------|
| **auto-increment** | 작음(8B), 정렬됨 (인덱스 효율↑), 사람이 읽기 쉬움 | 단일 노드 의존 → 분산 어려움, 순차적 → 보안 노출 | 단일 DB OLTP |
| **UUID v4** | 분산 환경에서 충돌 없이 생성, 보안성↑ | 16B로 큼, **랜덤 → INSERT 시 인덱스 페이지 스캔 증가** | 분산, 외부 노출 |
| **UUID v7** | 시간 정렬 + 분산 가능. **v4의 INSERT 문제 해결** | 16B 크기 | **2026 권장** — 모던 분산 시스템 |
| **Snowflake ID** | 64bit 시간 정렬, 노드 ID 포함 | Twitter ID 알고리즘 별도 구현 필요 | 트위터·디스코드 등 매우 큰 분산 시스템 |
| **ULID** | UUID v7 과 유사한 시간 정렬, base32 가독성 | 인지도 낮음 | 외부 노출 ID로 인기 |

**UUID v4 의 INSERT 문제:**

- 랜덤 값 → 새 행이 인덱스 트리의 임의 위치에 삽입
- B+Tree 페이지 분할 빈발 → 디스크 I/O ↑
- MySQL InnoDB 에서 특히 영향 큼 (clustered index)

**UUID v7 의 해결:**

- 상위 비트가 timestamp → 시간순 정렬
- 새 행이 인덱스 끝에 추가 → 페이지 분할 없음
- v4 의 분산 장점 + auto-increment 의 정렬 장점

**실무 결정:**

- 단일 DB 내부 PK: **auto-increment** (가장 효율적)
- 외부 노출 / 분산 ID: **UUID v7** (또는 ULID)
- 큰 분산 시스템 (대용량 트래픽, 노드 추적 필요): Snowflake ID
- **두 키 동시 사용 패턴** — 내부 PK는 auto-increment, 외부 노출 ID는 UUID v7 컬럼 별도 (인덱스 추가)

**보안 관점:**

- auto-increment를 외부 URL에 노출하면 enumeration 공격 (예: `/order/1`, `/order/2`)
- 외부 URL은 반드시 unguessable ID (UUID v4 또는 v7)

</details>
